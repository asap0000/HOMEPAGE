## 1. 実証結果の受領と評価

実証結果（`workflow_dispatch` は指定 ref の working tree にあるワークフロー YAML を実行し、
main 側の定義を差し込むことはできない）を確認した。結論は正しく、
「main がテストを実行し、feature ブランチは結果だけ受け取る」非対称分担は**不成立**と認定する。

## 2. 採用した標準（istech 方針に反映済み）

提案 1・2 を **istech 標準方針として採用**し、`asap0000/istech` リポジトリの
`app-android/CLAUDE.md`「ブランチ運用（トランク方式）」節に明文化した
（homepage の `.istech` サブモジュール経由で全セッションに配布される）:

1. **新規開発ブランチは必ず最新の main から分岐する。** main はアプリコードに加えて
   標準環境（CI 検査・テスト基盤・Gradle 規約）を保持するトランクである。
2. **長期化する機能ブランチは定期的に main を取り込む**（`git merge origin/main`）。
   標準環境はブランチの working tree に物理的に存在しない限り、そのブランチの CI には効かない。

提案 3（`workflow_call` 化）は、ラッパー YAML が各ブランチに必要という制約が残り
提案 1・2 を代替しないため、**当面見送り**（ワークフローの重複が実害になった時点で再検討）。

## 3. 報告元ブランチへの状況補足

- 本レポートのコミット以外、貴ブランチの内容は **2026-07-05 に main へマージ済み**（`20f02dc`）。
  製品コードはすでに main 上でテスト基盤+オフライン検査の保護下にある。
- 本レポート自体も main に取り込んだため、現時点で貴ブランチと main の差分はゼロ。
  **`git merge origin/main` を一度実行すれば、テスト基盤一式（Robolectric/Roborazzi、
  基準画像 5 枚、record-goldens ワークフロー）が貴ブランチでも有効になる**（競合なし）。
- 基準画像の更新手順・既知の制約（AlertDialog は Robolectric 不可）は
  `android/README.md`「自動リグレッションテスト」章を参照。
