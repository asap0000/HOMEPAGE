# 検討結果：本人確認証の「正規原本の出力（印刷/提出）」機能の是非と安全設計

> 依頼書「設計検討依頼書：本人確認証の『正規原本の出力（印刷/提出）』機能の是非と安全設計」への独立検討回答。
> 本書は検討のみであり、コード変更は行っていない。最終判断は開発者に委ねる。
> 2 モデル並行検討の 1 本目として、他モデルの結論との突き合わせポイントを §10 に列挙した。

---

## 0. 結論サマリ

**成立する。ただし「原本を出力する機能」としてではなく、「透かし入り・既定部分マスクの『提出用複製』を生成する機能」として設計し直すことが成立条件である。**

- 無地の原本ピクセルは今後も**一切**端末外に出さない（.pcbak を除く現状維持）。出力されるのは常に、透かしを焼き込んだ派生物「提出用複製」のみ。
- これにより不変条件 N2「原本が端末外に出る経路を新設しない」は**文字通り**維持され、製品の核心保証はバックドア化しない。出力機能の追加ではなく、既存の「マスク版のみ書き出し可」原則の**第三の派生物クラス（提出用複製）の追加**として位置づけられる。
- 推奨最小構成：**A（直接印刷のみ）＋ C（認証・制限・透かし）＋ F（部分開示を既定）＋ E（自画面 FLAG_SECURE 維持）＋ 監査ログ（N5）**。B（一時 PDF＋共有）は実装しない。共有インテントは一切使わない。
- 最重要の技術的知見：**印刷は INTERNET パーミッションなしで可能**（印刷サービスは別プロセス・別アプリが担う）。ただしその裏返しとして、**印刷データはシステムの印刷スプーラと印刷サービスに渡った時点でアプリの保護範囲外になる**。「ネット送信が技術的に不可能」という現行の訴求は、印刷経路には適用できない。訴求文言の精密化が必須（§7）。

## 1. 核心的再フレーミング：「原本出力」は実装しない

依頼書 §3 の緊張は、「原本」という語を出力物に使う限り解消できない。解消する設計は一つ：

| | 現行 | 追加後 |
|---|---|---|
| 原本（無地・暗号化保存） | 端末外に出ない（.pcbak のみ例外） | **変更なし** |
| マスク版 | 共有ギャラリーへ書き出し可 | 変更なし |
| **提出用複製（新設）** | — | 透かし焼き込み・既定部分マスク・印刷経路のみ・都度認証・記録必須 |

「提出用複製」は次をピクセルに焼き込む（メタデータではなく画像自体に）：

- 目的表示：「本人確認書類の提出用複製・目的外利用不可」
- **提出先（ユーザー入力必須。空欄では出力不可）**
- 出力日
- 全面に斜め繰り返しの半透明透かし（切り抜き・消去を困難にするため、余白ではなく券面に重ねる）

これで得られるもの：(1) 流出時の**追跡性**（どの提出用にいつ出したか）、(2) 流出物の**悪用価値の低減**（透かし入りコピーは本人確認素材として再利用しにくい）、(3) N2 の字義どおりの維持、(4) 「無地の原本は絶対に出ません」という**より強い訴求**への転換。

## 3. 脅威シナリオ耐性表（推奨構成 A+C+E+F+H に対して）

| 脅威 | 耐性 | 説明 |
|---|---|---|
| **T1** 一時ファイル横取り | **◎** | 自アプリのストレージに一時ファイルを一切作らない。復号→透かし合成→PdfDocument 生成をメモリ上で行い、印刷フレームワークから渡される FD へ直接書き込む（§8）。共有インテント不使用のため受け渡し横取り面もない。残る面はシステム印刷スプーラ（下記 T6）。 |
| **T2** ユーザーがクラウド同期フォルダへ保存 | **△** | 印刷 UI の「PDF として保存」を API で無効化できないため、経路は残る（§5-R2）。緩和：①同意画面で明示警告 ②保存されるのは透かし入り複製のみで無地原本ではない。**この脅威は技術では塞げず、透かしで被害を減価する方針で受容する。** |
| **T3** 強要・のぞき見 | **△** | 技術的に解決不能。緩和：都度生体認証（強要側に操作の壁）、出力履歴（事後検知）、透かしの提出先記入（強要による出力物にも追跡性が残る）。 |
| **T4** 印刷経路が実はネット送信 | **○（正直な限定付き）** | 自アプリは INTERNET 非宣言のまま（印刷に不要、§8）。ただし印刷サービス（Default Print Service / Mopria / ベンダープラグイン）は**別アプリで、ネット権限を持ち得る**。標準の IPP 経路はローカル LAN だが、ベンダープラグインのクラウド印刷機能（例：メーカーのリモート印刷）を使えばインターネットを通る。**アプリ側から印刷サービスの通信を制御・検知する手段はない。** 緩和：同意画面と説明書で「自宅プリンタへの直接印刷（標準の印刷サービス）」を推奨し、クラウド印刷系を使わないよう案内。 |
| **T5** 画面録画・アクセシビリティ | **○** | 自アプリの全画面（プレビュー含む）は FLAG_SECURE 維持で録画不可。アクセシビリティは画像ピクセルを直接取得できない。**ただしシステム印刷プレビュー画面は印刷スプーラ（別アプリ）の画面であり、FLAG_SECURE の保護外**（§5-R3）。 |
| **T6** OS 機構経由の残留・復元 | **○** | allowBackup=false 維持、自アプリに一時ファイルなし。**残る残留点は印刷スプーラの内部保存**（ジョブ完了まで、失敗ジョブは滞留し得る）。スプーラはサンドボックス内（他アプリから読めない）だが、自アプリの管理外。PrintJob の状態を監視し、完了/失敗をユーザーに表示して滞留ジョブの削除を促すことは可能。 |
| **T7** 将来の OS 仕様変更 | **○** | 印刷フレームワークは API 19 から安定しており変更リスクは相対的に低い。共有インテント・Photo Picker・スコープドストレージを**一切使わない**構成のため、変化の激しい領域への依存がない。対策：リリース前チェックリストに「印刷経路の再検証」を追加（istech 方針の機械的担保と整合）。 |

B 案が不採用である理由を T で言うと：T1（受け渡し先アプリの持ち逃げ）、T2（受け取りアプリのクラウド同期）、T6（受け取りアプリ内残留）に**構造的に敗北**し、しかも「即時削除」は自分のコピーにしか効かないため安心感だけを与える（最悪の性質）。

## 4. 推奨案（最小構成）の詳細

### 出力フロー（1 件・1 回ごと）

1. 対象書類を開く（既定マスク表示）→「提出用に印刷」を選択
2. **書類種別プリセットの選択**（後述）→ 開示範囲を確認・調整（F）
3. **提出先を入力（必須）**、出力日は自動
4. **同意画面**（§7 の文言）→ チェック → **生体認証/端末 PIN**（復号キーを認証に紐付け、§8）
5. メモリ上で：復号 → 部分マスク適用 → 透かし焼き込み → PDF 化 → PrintManager へ
6. **監査ログに追記**（書類 ID・日時・提出先・出力種別）。出力履歴画面（H3）で閲覧可能
7. まとめ出力・複数件同時出力は提供しない（N3）。連続出力は毎回 4〜6 を繰り返す

### 書類種別プリセット（F の具体化。初心者向けの差別化価値でもある）

日本の提出実務には法令上のマスキング要件があり、F は「安全のための不便」ではなく**提出先の要求への適合機能**として提示できる：

- **健康保険証**：被保険者記号・番号・保険者番号・QR コードを既定マスク（2020 年 10 月施行の告知要求制限により、提出先はそもそもこれらを収集できない）
- **マイナンバーカード**：**裏面（個人番号面）は出力を既定でブロック**（番号法上、個人番号の提供を受けられる相手は限定される。誤提出は利用者を法的リスクに晒す）。表面のみ出力可
- **運転免許証**：本籍欄なし面のみ。番号マスクの要否は提出先次第のため任意トグル
- **通帳・取引報告書**：口座名義・金融機関・必要ページのみ等、矩形指定マスク

全面開示は「提出先が全面を要求している」旨の追加確認を挟んだ上で許可する（禁止はしない。禁止すると再びアプリ外の撮り直しへ流れるため）。

### 採用しないもの

- 共有インテント（ACTION_SEND / chooser）：一切使わない
- 自アプリ領域への PDF 書き出し・SAF 経由のエクスポート：実装しない
- D（Pro 限定）：ビジネス判断としては自由だが、**安全性の根拠として扱わない**こと

## 6. 「実装しない方がよい」と判断する境界（G が妥当になる条件）

以下のいずれかに該当するなら実装を見送るべき：

1. **透かしなし・無地の原本出力が要件になった場合**。§1 の骨格が崩れ、N2 に本物の穴が開く。この要求には応じない（機能ごと見送る方が良い）
2. **共有インテント/汎用ファイルエクスポートが必須要件になった場合**（コンビニ印刷対応を v1 必須にする等）。B 案の構造的弱点を抱え込む
3. **R2（PDF 保存→クラウド流入）を製品として受容できない場合**。この経路は技術で塞げないため、受容できないなら出力機能自体が成立しない
4. **リリースごとの印刷経路再検証（T7 対応）を継続する保守体制が持てない場合**。istech 方針は機械的担保を要求しており、担保なしの安全機能は方針違反になる
5. 同意フローが形骸化するほど利用頻度が高い設計にしてしまう場合（毎日何十枚も出す用途が想定されるなら、都度同意は無意味化する。本製品の用途では該当しない見込み）

## 7. UX/同意文言案

### 同意画面（出力の直前、認証の前）

> **提出用の複製を印刷します**
>
> - 出力されるのは、提出先・日付・「目的外利用不可」の透かしが入った**提出用の複製**です。**無地の原本が端末の外に出ることはありません。**
> - 印刷データは、お使いの**プリンタ環境（Android の印刷サービス）**に渡されます。ここから先は本アプリの保護の外になります。**ご自宅のプリンタへの直接印刷**をおすすめします。
> - 印刷画面で「**PDF 形式で保存**」を選ぶと、複製がファイルとして端末に保存されます。**クラウドと同期されるフォルダには保存しないでください。**
> - この操作は、アプリ内の**出力履歴**に記録されます。
>
> 提出先（必須）：［　　　　　　　　］
> ☐ 上記を理解しました
> ［キャンセル］［認証して印刷へ進む］

### 訴求文言の精密化（重要）

現行の売り「ネット送信が技術的に不可能」はアプリ本体には引き続き真だが、印刷機能の追加後は次のように言い分ける：

- ○「**本アプリ自体はインターネットに接続できません**（通信権限を持ちません）。印刷は、お使いの端末の印刷機能を通じて行われます」
- ○「**無地の原本は、いかなる場合も端末の外に出ません**。印刷されるのは透かし入りの提出用複製だけです」
- ×「印刷してもデータは一切ネットに流れません」（印刷サービス次第で偽になり得るため、この断言はしない）

## 8. Android 技術メモ

- **INTERNET 権限なしの印刷**：可能。`PrintManager.print()` はシステムの印刷フレームワークへの IPC であり、ネットワーク通信（IPP 等）は**印刷サービスアプリ**（Default Print Service / Mopria / ベンダー製）が自身の権限で行う。**本リポジトリの CI 検査（INTERNET 宣言禁止）とそのまま整合する**。追加権限は不要
- **ファイルを作らない実装**：`PrintDocumentAdapter.onWrite()` はシステムから `ParcelFileDescriptor` を受け取る。復号ビットマップ→透かし合成→`PdfDocument`→`writeTo(FileOutputStream(fd))` で、**自アプリのストレージに一切書かずに**スプーラへ渡せる。`onFinish()` でビットマップを明示的に `recycle()` し、鍵マテリアルと合わせて参照を破棄する
- **書き込んだ先はスプーラ**：FD の実体は印刷スプーラ（システムアプリ）の私有領域。ジョブ完了まで保持され、失敗ジョブは残り得る。`print()` が返す `PrintJob` で状態（COMPLETED/FAILED/CANCELED）を監視し、完了確認を UI に出す
- **FLAG_SECURE の限界**：効くのは自アプリの Window のみ。システム印刷プレビュー（印刷スプーラの Activity）には効かない。自アプリ側のプレビューは既存どおり FLAG_SECURE 下に置く
- **「PDF として保存」**：印刷 UI の組み込み出力先で、**アプリから非表示化・無効化する公開 API はない**。存在を前提に同意文言で扱う（§7）
- **認証との紐付け**：提出用複製の生成に使う復号は、`BiometricPrompt` + `CryptoObject` で**認証都度**に限定する（`setUserAuthenticationParameters(0, AUTH_BIOMETRIC_STRONG or AUTH_DEVICE_CREDENTIAL)` 相当）。「認証済みセッションで何枚でも」を作らないことが N3 の実装上の本体
- **確実削除について**：本構成では自アプリ側に削除すべき一時ファイルがそもそも発生しない（これが A 案を選ぶ最大の実装上の理由）。仮に将来ファイル生成が必要になっても、FBE 環境のフラッシュストレージでは「上書き消去」は保証にならない点に注意
- **監査ログ**：アプリ私有領域に追記型で暗号化保存。項目：書類 ID・種別・日時・提出先・開示範囲（プリセット名）・ジョブ結果

| 依頼項目 | 本書の該当箇所 |
