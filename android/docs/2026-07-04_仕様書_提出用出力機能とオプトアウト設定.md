## 1. 目的と設計原則

本人確認書類の「提出用複製」を、印刷経路に限定して安全に出力できるようにする。同時に、出力機能そのものを設定で無効化（オプトアウト）できるようにし、出力・設定変更・ログ操作のすべてを端末内監査ログに記録する。

**不変原則（検討書 N1–N5 の継承。実装で破ってはならない）**

- P1. `INTERNET` パーミッションを追加しない（印刷に不要。既存 CI 検査を維持）。
- P2. **無地の原本ピクセルは端末外に出さない。** 出力されるのは常に透かし焼き込み済みの「提出用複製」のみ。
- P3. 出力は 1 件・1 操作・都度認証。まとめ出力なし。共有インテント（`ACTION_SEND` / chooser）は一切使わない。
- P4. 出力経路は Android 印刷フレームワーク（`PrintManager`）のみ。自アプリのストレージに一時ファイルを生成しない。
- P5. 原本出力に関わるすべての操作（出力・失敗・設定変更・ログ削除）を監査ログに記録する。
- P6. `FLAG_SECURE`・`allowBackup=false`・既定マスク表示など既存の保護は全画面で維持する。

## 2. スコープ

**含む**：F1 提出用出力機能／F2 設定画面（新設）／F3 監査ログ（記録・閲覧・自動圧縮・削除）

**含まない（明示的スコープ外）**：コンビニ印刷対応、PDF ファイルエクスポート（SAF 含む）、共有インテント経由の出力、クラウド連携全般、ログの端末外バックアップ（.pcbak にログを含めるかは §9-D5）

## 3. F1：提出用出力機能（制限付き）

### 3.1 出力フロー（画面遷移）

```
書類ビューア（既定マスク表示）
 └ メニュー「提出用に印刷」…設定OFF時は非表示（§4.2）
    → (1) 書類種別プリセット選択
    → (2) 開示範囲の確認・調整（部分マスクプレビュー）
    → (3) 提出先入力（必須・空欄不可）＋出力日（自動・編集不可）
    → (4) 同意画面（文言は検討書 §7 を正とする）＋チェックボックス
    → (5) 生体認証 / 端末PIN（BiometricPrompt + CryptoObject、都度認証）
    → (6) システム印刷UIへ（PrintManager）
    → (7) 完了/失敗の結果表示 ＋ 監査ログ記録
```

- (5) の認証は**出力 1 回ごと**に要求する。認証済みセッションの使い回しを実装しない（`setUserAuthenticationParameters(0, AUTH_BIOMETRIC_STRONG or AUTH_DEVICE_CREDENTIAL)` 相当で復号キーを都度認証に紐付ける）。
- (6) 以降のジョブ状態は `PrintJob` で監視し、COMPLETED / FAILED / CANCELED を (7) とログに反映する。

### 3.2 書類種別プリセット

| プリセット | 既定マスク | 特記 |
|---|---|---|
| 健康保険証 | 被保険者記号・番号・保険者番号・QR を既定マスク | 告知要求制限の説明文を (2) に表示 |
| マイナンバーカード | **裏面（個人番号面）は出力ブロック**（選択不可・理由表示） | 表面のみ出力可 |
| 運転免許証 | 本籍欄なし面のみ。免許証番号マスクは任意トグル | |
| 通帳・取引報告書 | 矩形指定マスク（既存 MaskingEngine の応用） | |
| その他/全面開示 | マスクなし | 「提出先が全面を要求している」旨の追加確認を挟む |

### 3.3 透かし（提出用複製の生成要件）

- 画像ピクセルに焼き込む（メタデータ・注釈は不可）：①「本人確認書類の提出用複製・目的外利用不可」②提出先（入力値）③出力日。
- 全面に斜め・繰り返し・半透明で券面に重ねる（余白のみへの配置は不可）。判読性と券面情報の視認性を両立させる透過率はゴールデンテストで固定する。
- 透かしなしの出力パスをコード上に存在させない（デバッグビルド含む）。

### 3.4 実装制約

- 復号 → マスク適用 → 透かし合成 → `PdfDocument` 生成 → `PrintDocumentAdapter.onWrite()` の FD へ直接書き込み。**`filesDir` / `cacheDir` / 外部ストレージへの書き込み禁止。**
- `onFinish()` でビットマップを `recycle()` し、参照を破棄する。
- 出力関連の全画面（(1)〜(4)、プレビュー）に `FLAG_SECURE` を適用する。
- システム印刷プレビューが FLAG_SECURE 外であること、「PDF として保存」を無効化できないことは既知の受容済みリスク（検討書 §5 R2/R3）。同意画面の文言でカバーする。

## 4. F2：設定画面（新設）

### 4.1 画面構成

```
設定
├ 提出用の印刷
│  ├ トグル「提出用の印刷を許可する」（§4.2）
│  └ 説明文（機能の要約＋「無地の原本は端末の外に出ません」）
├ 出力履歴（→ F3 閲覧画面へ）
└ ログの管理
   ├ 保存状況の表示（詳細ログ件数／アーカイブ月数）
   └ 「出力履歴を削除」（§5.4）
```

- ブランド基調（ブルー `#3366FF`・ホワイト背景・ミニマル）。専門用語には補足を添える（初心者向け）。
- 設定値は暗号化ストレージ（EncryptedSharedPreferences または暗号化 DataStore）に保存。`allowBackup=false` により端末外へ出ない。

### 4.2 出力機能トグル（オプトアウト）

| 項目 | 仕様 |
|---|---|
| キー | `output.print.enabled : Boolean` |
| 既定値 | **ON（オプトアウト方式）**※§9-D1 の決定対象。初回の出力実行時には必ず同意画面を通るため、既定 ON でも無同意出力は発生しない |
| OFF → ON | **生体認証必須**＋機能説明の再表示 |
| ON → OFF | 認証不要（安全側への変更のため）。確認ダイアログのみ |
| OFF 時の挙動 | 書類ビューアの「提出用に印刷」メニューを**非表示**にする（無効化グレーアウトではなく非表示。機能の存在自体を隠し、のぞき見・強要経路を減らす） |
| 記録 | ON/OFF いずれの変更も監査ログに記録（変更後の値・日時） |
| フレーバー | Lite / Pro とも同一仕様（§9-D2 で Pro 限定に変更可能なようフラグは分離しておく） |

## 5. F3：監査ログ

### 5.1 記録イベント

| eventType | 発火点 | 主な付帯情報 |
|---|---|---|
| `OUTPUT_PRINT` | 印刷ジョブ投入時 | 書類ID・種別・プリセット名・開示範囲・提出先・ジョブ結果（完了/失敗/取消。結果確定時に更新） |
| `OUTPUT_BLOCKED` | ブロック該当操作（例：マイナンバー裏面の出力試行） | 書類ID・理由 |
| `SETTING_CHANGE` | 出力トグル変更時 | 変更後の値 |
| `LOG_DELETE` | ログ削除の実行時 | 対象期間・削除件数 |
| `LOG_COMPACT` | 自動圧縮の実行時 | 対象月・件数 |

### 5.2 保存形式

- アプリ私有領域に**追記型**で保存し、既存 CryptoManager（AES-256-GCM × Keystore）で暗号化する。個別エントリの編集・個別削除の API は作らない（改ざん耐性）。
- エントリ構造（JSONL 相当）：`id / timestamp / eventType / documentId? / documentType? / preset? / destination? / jobResult? / detail?`
- 提出先はユーザー入力の生値を保存する（追跡性の本体）。ログ自体が機微情報になるため、閲覧画面も生体認証の後ろに置く。

### 5.3 自動圧縮（ストレージと見通しの両立）

- **詳細ログ**：直近 12 か月分または直近 1,000 件（いずれか大きい方）を生のまま保持。
- **アーカイブ**：それより古いエントリは、アプリ起動時の判定で**月単位に自動ロールアップ**する。月次アーカイブは gzip 圧縮 → 暗号化して保存し、詳細ファイルから除去する。
- 履歴画面では、アーカイブ月は「2025年6月：出力3件・設定変更1件」のような**集計行**で表示し、タップで展開（復号・伸長）して全文閲覧できる。
- 圧縮はユーザー操作・設定を要しない完全自動とし、実行の事実を `LOG_COMPACT` として記録する。**圧縮は削除ではない**（情報は失われない）ことを実装上保証する。

### 5.4 ログ削除（摩擦の調整）

設計方針：**「削除は重すぎず・軽すぎず、ただし削除の事実だけは絶対に残る」**。

- 提供する操作は**一括削除のみ**（全期間、または月単位の期間指定）。個別エントリの削除は提供しない。
- フロー：説明表示（下記文言）→ **生体認証（1回）** → 確認ダイアログ（対象件数表示・1回）→ 実行。パスフレーズ入力・待機タイマー等の追加障壁は**設けない**（無用な摩擦の排除）。
- 実行後、`LOG_DELETE` エントリ（日時・対象期間・件数）を**新規に記録**する。これにより、ログを空にしても「いつ・どの範囲を削除したか」だけは必ず残る。この挙動は削除前の説明文で明示する：

  > 出力履歴を削除します。削除した内容は復元できません。
  > なお、「履歴を削除した」という記録（日時と件数のみ）は残ります。

- `LOG_DELETE` エントリ自体も次回の一括削除の対象にはなる（そのときはまた新しい削除記録が残る）。「完全な無痕跡化」は仕様として提供しない。

### 5.5 リスクヘッジとしての限界（明記事項）

本ログは完全オフラインの端末内データであり、開発者側に自動送信されることはない（オフライン厳守）。したがって開発者のリスクヘッジとしての効力は、**紛争時に端末上のログを確認できた場合**に「ユーザー自身が認証・同意の上で出力した／機能を有効化した」ことを示す範囲にとどまる。第一義のリスクヘッジはあくまで**同意画面の文言と操作フローの設計**（無同意では出力が構造的に不可能であること）であり、ログはその補強である。プライバシーポリシー・利用規約に「出力履歴は端末内にのみ保存され、開発者は取得しない」旨を記載すること。

## 6. 非機能要件

- 出力フロー全体（認証除く）の追加待ち時間は 2 秒以内を目安（透かし合成・PDF 生成）。
- ログ書き込みは出力処理をブロックしない。ただし `OUTPUT_PRINT` の記録は**印刷ジョブ投入前に**確定させる（記録失敗時は出力を中止する。ログなし出力を許さない）。
- 自動圧縮は起動をブロックしない（バックグラウンド実行。WorkManager 使用可、ネットワーク制約は一切付けない）。

## 7. 受け入れ基準（テスト観点）

1. `AndroidManifest.xml` に `INTERNET` が存在しない（既存 CI で担保）。
2. 出力トグル OFF のとき、書類ビューアに印刷メニューが表示されない。OFF→ON に生体認証が要求される。
3. 出力フローで、認証・同意チェック・提出先入力のいずれかを欠くと印刷に到達できない。
4. 出力処理の実行中および完了後、`filesDir` / `cacheDir` / 外部ストレージに新規ファイルが生成されていない（計測テスト）。
5. 出力 PDF に透かし（目的表示・提出先・日付）が焼き込まれている（ゴールデンテスト）。透かしなしで出力できるコードパスが存在しない。
6. マイナンバーカード裏面プリセットで出力がブロックされ、`OUTPUT_BLOCKED` が記録される。
7. 出力・設定変更・削除・圧縮の各操作で対応する監査ログエントリが生成される。ログ記録に失敗した場合、出力が中止される。
8. 13 か月以上前のエントリが月次アーカイブへ移動し、展開閲覧で内容が一致する（圧縮の可逆性）。
9. ログ一括削除後、`LOG_DELETE` エントリのみが残存する。
10. 出力関連の全自アプリ画面でスクリーンショットが遮断される（FLAG_SECURE）。
11. 静的検査：印刷関連モジュールに `ACTION_SEND` / `Intent.createChooser` / `FileProvider` 経由の外部共有が存在しない（CI に grep 検査を追加）。

## 8. CI・機械的担保への追加（istech 方針準拠）

- 既存の「INTERNET 宣言禁止」「ネットワーク系ライブラリ混入禁止」検査を維持。
- 追加：出力モジュールに対する共有インテント検査（§7-11）をビルド時チェックに組み込む。
- リリースチェックリストに「OS アップデート後の印刷経路再検証」を追加（検討書 T7 対応）。

## 9. 要決定事項（実装前に確定すること）

| # | 論点 | 本書の暫定値 | 代替案 |
|---|---|---|---|
| D1 | 出力トグルの既定値 | **ON（オプトアウト）** | OFF（オプトイン）。「初期状態では印刷すらできない」をブランド訴求に使える。都度同意があるため安全性の実差は小さく、訴求と摩擦のトレードオフ |
| D2 | Lite / Pro の差別化 | 両方に搭載 | Pro 限定（ビジネス判断。安全根拠にはしない） |
| D3 | 詳細ログ保持境界 | 12 か月 / 1,000 件 | 期間・件数の調整 |
| D4 | 削除の粒度 | 全期間＋月単位 | 全期間のみ（さらに簡素化） |
| D5 | .pcbak バックアップにログを含めるか | **含めない**（ログは端末ローカル限り） | 含める（機種変更でも履歴継続。ただしログの端末外複製が生じる） |
