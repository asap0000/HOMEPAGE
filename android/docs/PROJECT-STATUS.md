## 1. プロダクト概要
プライバシー重視のカメラアプリ（Android / Kotlin + Jetpack Compose）。1コードベースから2フレーバー：
- **Lite**（`com.privacycamera.lite`）: 無料・お試し。保存上限あり・自動モザイクのみ。
- **Pro**（`com.privacycamera`）: 有料想定。無制限・取り込み・高度マスキング等。
- 出し分けは `BuildConfig.IS_PRO`（`com.privacycamera.Tier`）が唯一の判定点。`Tier.LITE_SAVE_LIMIT = 30`。
- minSdk 26 / targetSdk 35。
- 設計思想（＝売り）: **INTERNET権限なし**（ネット送信が技術的に不可能）、原本は**AES-256-GCM × Android Keystore**で暗号化、表示は既定でマスク、`FLAG_SECURE`、`allowBackup=false`。

## 3. ビルド・署名・リリース運用
- **署名鍵**: PKCS12 keystore（`app/release.keystore.gpg` に GPG対称暗号で格納）。**今後ずっと固定維持**（鍵が変わると上書き更新不可＝データ消失）。
  - 署名鍵 SHA-256（ADC等で使用・公開可）:
    `57:D6:89:DF:BD:95:87:15:92:26:F2:44:6D:61:06:6B:45:9D:93:96:A9:44:93:FE:A9:07:F2:54:EE:DD:4C:36`
  - PKCS12は鍵PW=ストアPWに強制されるため、gradleは `keyPassword = RELEASE_STORE_PASSWORD` を使用（`RELEASE_KEY_PASSWORD` の値に非依存）。
  - 秘密値（GPGパスフレーズ・ストアPW・エイリアス`upload`・鍵PW）は **GitHub Actions Secrets** に登録済み（`KEYSTORE_PASSPHRASE` / `RELEASE_STORE_PASSWORD` / `RELEASE_KEY_ALIAS` / `RELEASE_KEY_PASSWORD`）。**この文書には値を書かない**（パスワードマネージャ等で別管理）。
- **versionCode/versionName 自動採番**（`app/build.gradle.kts` + `.github/workflows/release.yml`）:
  - `versionCode = 1000 + github.run_number`（単調増加）、`versionName = タグから先頭v除去`。
  - ローカル/デバッグCIは `1` / `0.0-dev` にフォールバック。
  - 注意: beta v0.1〜v0.3 は versionCode=1。次リリースは ~1011 になり >1 なので上書き更新可。
- **リリースタグは毎回上げる（重要ルール）**: 修正版を出すときは**必ず新しいバージョンタグ**にする（例: `v0.3.2-beta` → `v0.3.3-beta`）。
  - 理由: 同一タグで上書き再発行すると **ダウンロードURLが不変**のため、ブラウザ/CDNが**旧APKをキャッシュ配信**し「更新したのに反映されない／インストールが一瞬で終わる」事故が起きる。versionName据え置きだと実機で更新可否も目視確認できない。
  - **運用**: ユーザーが「同じバージョン番号のまま（上書きで）出して」と指示した場合は、**そのまま実行せず上記リスクを注意勧告し、番号を1つ上げるよう促す**。それでも据え置き希望なら最終判断はユーザーに委ね、指示どおり実行する。
  - 補助: アプリのメニュー最下部（保護フォルダのドロワー）に `BuildConfig.VERSION_NAME`（ティア接尾辞付き, 例 `0.3.3-beta-pro`）を表示済み。実機で更新反映を確認できる。
- **CI**: `android.yml`（push連動でdebug/release両ビルド＝健全性確認）、`release.yml`（署名APKをGitHub Releaseへ）。
- **リリースの回し方（重要）**: この実行環境では **git tag の push が 403 で不可**。タグの代わりに **workflow_dispatch** で起動する：
  - GitHub MCP: `mcp__github__actions_run_trigger`（`method: run_workflow`, `workflow_id: release.yml`, `ref: <作業ブランチ>`, `inputs: {tag: "vX.Y-beta"}`）。
  - これで `ref` のコード（＝作業ブランチHEAD）をビルドし、`action-gh-release` がタグ付きReleaseを作成・APK添付。
- リポジトリスコープ: GitHub MCP は `asap0000/homepage` のみ。

## 4A. バックアップ/復元の実運用仕様（実機検証済み・v0.3.6〜v0.3.7）
> 「再調達不可能な写真」を扱う前提で、細かく作り込んだ最重要機能。コード: `data/BackupManager.kt`, `data/SecurePhotoStore.kt`, `viewmodel/PhotoViewModel.kt`, `crypto/BackupCrypto.kt`。

- **暗号化バックアップ**（Pro・`.pcbak`）: パスフレーズ→PBKDF2-HMAC-SHA256(12万回)→AES-256-GCM。ヘッダ `MAGIC "PCB1" | salt(16) | iv(12)`、本体はGCM暗号文（tag込み、ヘッダをAADで認証）。フォーマットは v0.3.x で不変。
- **復元の不具合修正（重要）**: GCM復号を `CipherInputStream` で行うと**複数ブロック＝大きめのファイルで壊れる**既知の問題。小さいバックアップは通り、18枚で「パスフレーズ違い/破損」失敗が発生。→ **全暗号文を読み込み一括 `doFinal`**（tag検証＝完全性/認証を兼ねる）に変更。`largeHeap` 付与。**旧v0.3.5で作った壊れファイルは復元不可（作成時に切り詰め破損）だが原本は端末内に健在で損失なし**。
- **書き出し直後の復元検証**: 書き出し後にその場で読み戻し `verifyEncrypted` で復号確認。**復元可能を確認できたときのみ成功表示**（「復元できることを確認しました」）。失敗時は「信頼できません・作り直して」。壊れバックアップを掴む事故を根絶。
- **重複判定＝マージ・ポリシー（削除トゥームストーン）**: 復元は「**ライブに既存**」＋「**この端末で削除済み(UUIDを`deleted_uuids.json`に永続記録)**」をスキップし、それ以外を取り込む。
  - **同一機器**: 削除の事実を保持 → 削除済みは**復活させない**（自己削除ゾンビ復活しないポリシー）。
  - **別機器**: トゥームストーン空 → 削除の事実を継承しないので**復元して戻す**。
  - トゥームストーンは `delete()` で付与・ゴミ箱`restore()`で解除。
- **実機検証OK**: 主機→別機20枚=復元20/スキップ0。別機(既存3+20=23)→主機=復元3/スキップ20（相互マージ成功）。同一ファイル再選択=復元0/スキップ23（重複なし）。
- **復元中インジケータ**: 復号・取り込み中は「復元中…」＋サークルの遮断オーバーレイ表示。
- 既知の制約: 復元はカスタムマスク仕様を持ち越さない（原本のみ搬送→復元後は全面モザイク）。Lite移行書き出しは平文。
- **上位互換ポリシー**（明文化・2026-07-04）: `docs/2026-07-04_バックアップ上位互換ポリシー.md` 参照。要点＝**下位互換MUST（新版は過去の全 .pcbak を復元可能）／上位互換ベストエフォート（未対応版は破損でなく「未対応」と明確拒否）／manifestは加算則で上下互換**。バックアップにログ・画像来歴は**含めない**（D5決定）。実機上位互換テストは v0.3.6以降の健全バックアップを基準に実施。

## 4B. 提出用出力機能（印刷）— 設計検討・決定事項（実装未着手）
> 別モデルで脅威モデリング済み。仕様書（ユーザー保有）＋検討書を今後 `android/docs/` に正典化予定。**現時点で実装コードは無し**。
- **目的**: 本人確認証を「提出用複製」として**印刷経路(PrintManager)限定**で出力（コンビニ再撮影の解消）。原本ピクセルは出さず、**透かし＋本人が手動マスクした複製のみ**を印刷アダプタのFDへ直書き（一時ファイル不可）。
- **不変原則**: INTERNET権限追加なし／共有インテント(ACTION_SEND)不使用／1件1操作・都度認証／FLAG_SECURE等の既存保護維持。
- **決定事項**:
  - ① **内容解析なし・テンプレートなし**堅持。書類種別プリセットの自動マスクは**撤廃**。マスクは**本人が手動範囲指定**（既存 `MaskEditScreen`/`MaskingEngine` 再利用）。マイナンバー裏面の技術的ブロックは不可→**同意画面での注意喚起**に格下げ（本人のカテゴリ自認に基づく表示のみ）。
  - ② 都度認証は**アプリ層 `BiometricPrompt` ゲート**（reveal と同方式。現行マスター鍵は認証非拘束のため CryptoObject 拘束は採らない）。
  - ③ 監査ログは追記型。`OUTPUT_PRINT`（投入）と `OUTPUT_RESULT`（結果）を**別エントリ**で積む（既存エントリを更新しない＝改ざん耐性）。
  - **D1** 出力機能は**隠し機能**（開発者オプション風）。秘匿解除は**パッチで継承・アンインストールでリセット**（`allowBackup=false`＋暗号化prefsに `settings.revealed`保存で自動成立）。「再び隠す」操作も用意。
  - **D2** **Pro版のみ**（設定画面もPro限定。フラグ分離）。
  - **D3/D4** ログ保持=12か月/1000件、削除=全期間＋月単位（現行案）。
  - **D5** バックアップにログ・画像来歴を**含めない**（確定）。
- **次工程**: 仕様書 v1.1（上記反映）＋検討書を `android/docs/` にコミット→ 実装計画（影響ファイル/テスト=受け入れ基準§7）作成→ 実装。

### STORES（EC）
- ダウンロード販売: ≤1GB/件、購入で自動配信（購入ゲート）。自作ソフトの販売可（違法/著作権侵害のみ禁止）。

- 既存購入者: **会員ログイン購入なら購入履歴から無期限で再DL**可。→ 無償更新は「ファイル差し替え＋再DL＋告知」or「¥0クーポン」。
- **「所在地・電話番号 非公開」機能**（個人事業主向け）: STORES社の住所/電話が代理表示され、自宅住所/電話を隠せる。ただし **事業者氏名＝本名は表示必須**（屋号のみ不可）。氏名も隠すなら**法人化**。
- 買い手側: デジタルDLは配送先住所が不要 → 買い手の個人情報露出は最小（メルカリ的な相互住所交換は起きない）。

### Android Developer Console（ADC）＝開発者確認（Play掲載とは別）
- 個人で登録可（**D-U-N-S不要**）。**Limited**（無料・≤20台・政府ID不要・2026/8開始）／**Full**（$25・政府ID＋住所証明）。
- **パッケージ登録はアプリごと**（SHA-256登録＋所有証明APLをアップ）。Pro/Lite=2件。
- 強制: 4か国2026/9 → **global 2027**（日本は概ね2027）。効果は「未登録＝身元不明アプリのブロック回避」。

#### 最新動向（2026-07 時点。実機での目視 ＋ 公開情報で確認）

### 戦略
- サイドロード摩擦は「摩擦を厭わない＝ストア非依存/プライバシー志向」層をふるい分ける。現アプリ（脱Google・ローカル完結）はこの層と理想的に合致。**「Google圏外で完結」を売りに**。
- 身元確認は1回の固定費 → **1つの検証済み身元の下に同系統の製品ラインを束ねる**ほど割に合う（暗号メモ/ローカル金庫/オフライン書類スキャナ等の横展開候補）。
- 専用Googleアカウント分離運用は可。最大リスクは**復旧/ロックアウト（単一障害点）**→ 復旧情報・2FA・バックアップコードを堅牢に。署名鍵とGitHubはGoogleアカウントと独立。
- 個人事業主はGoogleの「個人（実名）」区分を変えない（法人化＝organization＋D-U-N-S）。税務/銀行/屋号表示に有用。
