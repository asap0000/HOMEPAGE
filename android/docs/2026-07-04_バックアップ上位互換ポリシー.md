# バックアップ上位互換ポリシー（暗号化バックアップ .pcbak）

| 項目 | 内容 |
|---|---|
| 対象 | PrivacyCamera の暗号化バックアップ（`.pcbak`）／復元 |
| 目的 | 端末間・**アプリ版間**で、撮影原本＋メタデータを安全に持ち運び・復元できること |
| コード | `data/BackupManager.kt`（export/importEncrypted/verifyEncrypted）、`crypto/BackupCrypto.kt` |
| 状態 | v1.0（2026-07-04 策定） |

> 決定の背景：D5 の議論で「バックアップは何のためか／上位互換性が欲しい」が論点になった。ログや画像来歴は**含めない**と決めたため、バックアップが運ぶのは「原本＋最小メタデータ」に限定される。その前提で、版が進んでも復元できる保証を明文化する。

---

## 1. バックアップが運ぶもの / 運ばないもの
- **運ぶ**：撮影原本（JPEG バイト列）、および最小メタデータ（`uuid` / `createdAt` / `caption` / `category`）。
- **運ばない（意図的）**：
  - 監査ログ（出力・アクセス等のイベント）→ **端末ローカル限り**（D5-1）。
  - 画像の編集来歴（トリミング等の履歴）→ 再利用性が低くコスト過大のため**採用しない**（D5-2）。
  - カスタムマスク仕様（復元後は全面モザイクに戻る。既知の制約）。

## 2. コンテナ形式（現行 = PCB1）
```
MAGIC "PCB1"(4) | salt(16) | iv(12) | AES-256-GCM 暗号文（manifest JSON + 各写真の JPEG。tag込み）
ヘッダ(MAGIC|salt|iv) は GCM の AAD として認証。
manifest JSON は { version, createdAt, count, photos:[{uuid,createdAt,caption,category,file}] }。
```
- 復号は**全体を一括 doFinal**（GCM の完全性検証＝改ざん/破損検知を兼ねる）。書き出し時は**その場で読み戻して復号検証**（`verifyEncrypted`）してから成功表示。

## 3. 互換性の保証レベル（本ポリシーの核心）
- **下位互換（新しいアプリ ← 古いバックアップ）＝ 保証する（MUST）。**
  新版は、過去に発行した**すべての** `.pcbak` を復元できること。importer から旧版リーダを削除しない。これが最重要保証（＝ユーザーの原本は将来も必ず戻せる）。
- **上位互換（古いアプリ ← 新しいバックアップ）＝ ベストエフォート（不保証）。**
  新版で作ったバックアップを旧版が開けるかは保証しない。ただし**壊れた復元を返さない**こと：旧版が扱えない版なら、`NotABackup`／「未対応バージョン」で**明確に拒否**する（黙って部分復元・誤復元しない）。
- **メタデータ拡張＝双方向互換（JSON の加算則）。**
  manifest はフィールドの**追加のみ可**。既存キーの意味変更・削除・型変更は**禁止**。読み手は「未知フィールドは無視／欠損はデフォルト補完」。これによりメタデータ変更は上下互換を自然に満たす。
- **JPEG ペイロード＝不変。** 原本バイト列は変換・再エンコードしない。

## 4. バージョニング規則
- **manifest の `version` を上げる**：メタデータの**加算的変更**時（上下互換を維持できる範囲）。リーダは version で分岐しつつ、未知フィールドは無視。
- **コンテナ MAGIC を上げる（例：`PCB2`）**：暗号方式・レイアウトなど**破壊的変更**時のみ。importer は **PCB1 と PCB2 の両リーダを保持**（下位互換 MUST）。旧版は未知 MAGIC を `NotABackup` として安全に拒否。
- 破壊的変更は**最終手段**。まずは manifest 加算で吸収できないかを検討する。

## 5. 実装・テストへの要求（コード着手時）
- importer は **version / MAGIC を明示的に読み、対応表で分岐**する（暗黙の単一版前提にしない）。
- 未対応版は**破損ではなく「未対応バージョン」として区別**して通知（ユーザーが「古いアプリで新しいバックアップを開こうとした」と分かる文言）。
- **クロス版復元テスト行列**を CI/手動で維持：`{各過去版で作成した .pcbak} × {最新アプリ}` が復元成功すること（下位互換 MUST の担保）。

## 6. 既知の経緯（誤解の払拭）
- 「**同一ビルド間でしか復元できない**」ように見えた事象は、**形式非互換ではなく**、旧 `CipherInputStream` による GCM 復号の**破損バグ**が主因（v0.3.6 で一括 `doFinal` に修正済み。別機種間の復元成功を実機確認済み）。
- したがって**上位互換は現実的に達成可能**。実機での上位互換テストは、**v0.3.6 以降で作成した健全なバックアップ**を基準に行うこと（破損していた v0.3.5 生成ファイルを基準に使わない）。

## 7. 改訂履歴
